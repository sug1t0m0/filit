# filit 要求・要件 (暫定)

> **Status:** WIP / 暫定ドラフト
> **Last updated:** 2026-07-08
> 初期の思考整理用ドキュメント。実装と共に随時更新する。

## 1. 背景

- 手元の Claude Code は使用量に制約がある。広範な調査・要約・大量文脈の読み込みは、ブラウザで使える Gemini (Gemini in Chrome) 側に寄せたい。
- Gemini in Chrome は**現在開いているタブの本文を文脈**として利用できる。最大10タブまで共有可能。ページ本文は UTF-8 バイト数で切り詰められる (公式に上限値は未公開)。
- 既存の類似ツール (`code2prompt`, `files-to-prompt`, `repomix` 等) はクリップボードやファイル出力が中心で、**「注釈を書き込んでLLMに投げる場」**にはなっていない。

## 2. 動機・要求 (Goals)

1. AIに相談したい複数ファイルの内容を **選んで・ブラウザ1タブに束ねて・そのままLLMに渡す** ワークフローを最短動線で実現する。
2. **選んだファイルに行/範囲単位で「ここに注目して」「これは何？」の注釈を書き込み、合成Markdownで該当行の近くに差し込める**。単なるバンドラーではなく、**AIへの問いかけを組み立てる場**として機能する。
3. 「入れすぎ / 入れ足りない」を可視化 (UTF-8 バイト数) して、トークン予算の勘所を育てる。
4. AI調査結果を叩き台に設計を詰める作業の入口として機能する。
5. 完全にローカルで動作し、リポジトリ内容が外部に送信されない。

## 3. 想定ユースケース

- **U1**: 特定ディレクトリのコード数ファイルを選び、Gemini に「この設計の妥当性を評価して」と相談する。
- **U2**: git で変更したファイル群だけを束ね、Gemini に「PR説明文の下書き」を書かせる。
- **U3**: 特定関数の `L20-L40` に「この境界は妥当?」とコメントを付け、Gemini に投げてピンポイントで回答をもらう。
- **U4**: 調査結果 Markdown + 関連コードを束ね、AI と設計の叩き台を詰める。
- **U5**: 特定機能に関わるファイルセットを名前付きで保存し、後で再利用する。

## 4. 要件

### 4.1 MVP (必須)

| ID  | 要件                                                                                                                                                                                 |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| R1  | CLI: `filit <paths>` で起点となるスコープを指定できる (ファイル・ディレクトリ・グロブ混在)                                                                                           |
| R2  | ブラウザ上でスコープ配下のファイルツリーを表示し、チェックボックスで選択できる                                                                                                       |
| R3  | 選択中のファイル群を **UTF-8 バイト数** としてライブ表示する                                                                                                                         |
| R4  | ファイルの**特定の行または行範囲にコメントを追加**できる (difit ライクな UI)                                                                                                         |
| R5  | コメントは対象リポジトリ直下の `.filit/comments.json` に **サーバ側で永続化**される                                                                                                  |
| R6  | 「タブに開く」ボタンで、選択したファイル + 該当コメントを **プレーン Markdown** としてまとめた別タブを開ける。コメントは該当行の近くに `[注釈 L2-4: ...]` のような形式で差し込まれる |
| R7  | 起動時に自動でブラウザを開く                                                                                                                                                         |
| R8  | 完全ローカル動作。ファイル内容もコメントもサーバ外に送信しない                                                                                                                       |

### 4.2 MVP 後 (優先度順)

| ID  | 要件                                                                               |
| --- | ---------------------------------------------------------------------------------- |
| R9  | git 連携: 「変更ファイル」「staged」「ブランチとの diff」などのクイックフィルタ    |
| R10 | ファジー検索でツリー内のファイルを絞り込む                                         |
| R11 | 選択セット (どのファイルにチェックを入れていたか) を名前付きで保存して再利用できる |
| R12 | 10タブ分割モード: サイズが大きい場合、複数タブに自動分配                           |
| R13 | 出力順の並び替え (ドラッグ) — 文脈に効くため                                       |
| R14 | ファイル単位の追加コメント (ファイル冒頭・末尾の指示ブロック)                      |
| R15 | コンテンツハッシュベースのコメント再アンカリング (ファイル変更後の追従)            |

### 4.3 非機能

| ID  | 要件                                                                                       |
| --- | ------------------------------------------------------------------------------------------ |
| NF1 | Node.js >= 20 で動作する                                                                   |
| NF2 | `npx filit` で試せる                                                                       |
| NF3 | 外部への通信を一切行わない (テレメトリなし)                                                |
| NF4 | 生成される合成ページはシンプルな Markdown + `<pre>` を基本にし、装飾でバイト数を圧迫しない |
| NF5 | MIT License                                                                                |
| NF6 | コメントの行アンカリングは MVP ではベストエフォート (ファイル変更でズレる可能性は受容)     |

## 5. スコープ外 (Non-goals)

- 特定 LLM サービスへの直接 API 連携 (**ブラウザタブ経由**が第一級の連携方式)。
- diff レビュー機能 (差分レビューは [difit](https://github.com/yoshiko-pg/difit) を使う)。
- リアルタイム協業・マルチユーザ機能。
- コメントのバージョン管理・履歴・スレッド。
- クラウドホスティング前提の機能。

## 6. 類似ツールとの位置づけ

| ツール            | 出力先                          | 注釈UI                                 | 主用途                                  |
| ----------------- | ------------------------------- | -------------------------------------- | --------------------------------------- |
| `code2prompt`     | クリップボード / ファイル       | なし                                   | プロンプト用テキスト生成                |
| `files-to-prompt` | 標準出力                        | なし                                   | 同上                                    |
| `repomix`         | ファイル                        | なし                                   | 単一ファイルへのリポジトリ束ね          |
| `difit`           | ブラウザ (diffレビューUI)       | 行単位あり (diff対象)                  | ローカルdiffのGitHub風レビュー          |
| **`filit`**       | **ブラウザタブ (Markdown合成)** | **行/範囲単位あり (素のファイル対象)** | **タブ本文経由でLLMに文脈と問いを渡す** |

## 7. コメント永続化の設計 (暫定)

- 保存先: 対象リポジトリ直下の `.filit/comments.json`
- `.filit/` を `.gitignore` に自動追加は**しない** (共有したい人が git 管理できる余地を残す)。README で言及する。
- スキーマ (仮):

```json
{
  "version": 1,
  "comments": [
    {
      "id": "uuid",
      "file": "src/auth.ts",
      "startLine": 20,
      "endLine": 40,
      "body": "この境界は妥当?",
      "createdAt": "2026-07-08T00:00:00Z"
    }
  ]
}
```

## 8. 合成 Markdown の書式 (暫定)

イメージ:

```
## src/auth.ts

    1: async function withdraw() {
    2:   [注釈 L2-4: ここのロックは何を守ってる?]
    3:   await lock()
    4:   ...
    5: }
```

- 行番号付き `<pre>` にするか、素の Markdown コードブロック + 注釈行にするかは実装で検討 (Gemini in Chrome の解釈のしやすさで判断)。

## 9. 未決事項 (Open Questions)

- **Q1**: 実行時のデフォルトポート番号 (仮: `4967` — difitの4966に隣接)。
- **Q2**: Gemini in Chrome の実効的な UTF-8 バイト上限値 (実測要)。警告閾値のデフォルト。
- **Q3**: 10タブ分割モードの分配戦略 (単純順序保持 / セマンティック分割 / ファイル境界優先)。
- **Q4**: 選択セット (R11) はリポジトリ横断で管理か、リポジトリごとに分けるか。
- **Q5**: `.filit/` を `.gitignore` に含めるべきかの推奨は?
- **Q6**: コメント合成書式 (§8) のベスト形状 — 行番号付き `<pre>` vs. Markdown コードブロック + 別行注釈。
- **Q7**: コメントの行アンカリングのドリフト対策 (再アンカリング) を実装するタイミング。
